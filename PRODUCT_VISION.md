# Spark (星火) - 產品願景與定位

> **「Ignite relationships that multiply.」**
> **「點燃會自我增殖的人脈關係。」**

---

## 一、產品本質

### 我們不是什麼

- ❌ 不是聯絡人管理工具（那是通訊錄）
- ❌ 不是 CRM 系統（那是銷售漏斗）
- ❌ 不是 LinkedIn（那是職位搜尋）
- ❌ 不是電子名片（那是資訊交換）

### 我們是什麼

✅ **Spark = 人脈資本的 AI 戰略顧問**

我們幫助業務人員做三件事：

1. **DISCOVER（發現）** - 發現你的多面性價值
2. **CULTIVATE（育人）** - 用 AI 匹配聯絡人的需求，創造有價值的互動
3. **MULTIPLY（擴張）** - 讓你的分享自然擴張到二度人脈

---

## 二、目標客群

### 主要客群

**業務開發人員**（汽車銷售、直銷、保險業務、房地產經紀等）

### 客群畫像

```
┌─────────────────────────────────────┐
│  張業務（保險經紀）                 │
│  - 手機裡有 500+ 聯絡人             │
│  - 每天新增 3-5 位新客戶            │
│  - 問題：不知道該聯絡誰、什麼時候聯絡│
│  - 痛點：錯過客戶的重要時刻         │
│  - 目標：讓陌生客 → 熟客 → 轉介紹   │
└─────────────────────────────────────┘
```

### 核心洞察

**他們不是沒有人脈，而是不知道如何經營人脈。**

---

## 三、核心痛點

### 痛點一：單一印象陷阱

**現況：**
- 聯絡人對業務人員的印象 = 「那個賣保險的/車的/房子的」
- 這樣的印象太單薄，不會讓人主動聯繫

### 痛點二：被動且無效的互動

**現況：**
- 只在被動問候：生日祝福、升遷問候、結婚紀念日
- 這些互動沒有價值，不會建立深度關係

### 痛點三：錯失潛在商機

**現況：**
- 潛在客戶就在聯絡人的好友名單中
- 但沒有機會讓他們看到你、認識你

### 痛點四：無法展現多面性

**現況：**
- 每個人都有豐富的人生經驗
- 但沒有一個平台可以系統化地展現這些價值

---

## 四、解決方案：人脈資本經營模式

### 核心哲學：進人、育人、留人

```
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  傳統思維（錯誤）                新思維（正確）              │
│  ├─ 我 = 保險業務                ├─ 我 = 多面向的有趣人類   │
│  ├─ 價值 = 賣保險給你            ├─ 價值 = 分享我的人生見解  │
│  ├─ 互動 = 被動問候生日          ├─ 互動 = 主動分享知識/故事│
│  └─ 目標 = 你買保險              └─ 目標 = 你覺得我有用，   │
│                                    然後介紹你的朋友給我      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 核心概念：物理性人生分享

**為什麼「物理性人生分享」是 AI 時代的護城河？**

| AI 能做的事 | 人類能做的事（AI做不到） |
|-------------|------------------------|
| 寫一篇關於登山的心得 | 分享**我**在上週六爬山時遇到暴雨，迷路3小時的**真實故事** |
| 生成 ESG 報告 | 分享**我**如何從 skeptic 變成 ESG 信徒的**心路歷程** |
| 列出咖啡推薦 | 分享**我**在巷弄裡發現那家咖啡店，與老闆聊天的**人情故事** |

**關鍵差異：**
- AI = **通用知識**
- 人 = **獨特經驗** + **情緒** + **不完美**

### 核心價值：讓好友知道你「有被利用的價值」

**「育人」的第一性原理：**

> 不只是問候，而是準備一個知識庫，讓好友知道你有什麼可以被利用的價值。

**拋棄單一專業印象，轉向多樣化的物理性人生分享**：
- 這才是 AI時代目前無法被取代的
- 這才是人與人可以快速建構被利用價值的地方
- 這才是業務人員建立深度關係的關鍵

## 七、核心功能架構

### 三大支柱

```
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                │
│  │ DISCOVER│ -> │ CULTIVATE│ -> | MULTIPLY│              │
│  │ 發現    │    │ 育人     │    │ 擴張    │               │
│  └─────────┘    └─────────┘    └─────────┘                │
│       ↓              ↓               ↓                     │
│                                                              │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                │
│  │你的     │    │AI 匹配  │    │社交影響 │                  │
│  │多面性   │    │互動     │    │力追蹤   │                  │
│  │價值庫   │    │策略     │    │         │                  │
│  └─────────┘    └─────────┘    └─────────┘                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 功能一：Vault（價值寶庫）

**目標：** 幫你發現並管理你的多面性價值

**功能：**
- 建立你的「價值知識庫」
- 每個知識領域包含多個「真實故事」
- 每個故事可轉化為「可分享的內容」
- AI 分析哪些故事最受歡迎

**用戶價值：**
> 「我不只是一個保險業務，我是登山者、咖啡狂、單親爸爸...這些都是我的資產。」

### 功能二：AI 談話策劃師

**目標：** 告訴你該聯絡誰、該說什麼

**功能：**
- 追蹤聯絡人在 LinkedIn/Facebook/IG 的動態
- AI 分析：哪些動態值得回應？
- AI 匹配：你的哪個故事最相關？
- AI 生成：具體的回應文案

**用戶價值：**
> 「我不用再猜該說什麼，Spark 告訴我：王小明剛分享 ESG 文章，匹配我的『從 skeptic 到 ESG 信徒』故事。」

### 功能三：社交影響力擴張器

**目標：** 讓你的分享自然擴張到二度人脈

**功能：**
- 追蹤你的社交互動影響力
- 分析：誰看到了你的分享？
- 分析：有多少新連結是透過互動產生的？
- 視覺化：你的關係地圖

**用戶價值：**
> 「我在王小明的文章下留言，結果他的好友陳美麗看到，主動連結我。這就是自然擴張。」

### 我們的技術哲學

#### 原則 1：降低摩擦力到接近零

**問題：** 為什麼大多數業務人員不使用 LinkedIn API 整合？

**傳統思維（Android 思維）：**
- 提供功能：LinkedIn API 整合
- 用戶責任：自己申請 Developer 帳號、等待審核、設定 OAuth
- 結果：90% 的用戶在步驟 2 就放棄

**Steve Jobs 思維（iPhone 思維）：**
- 理解用戶：業務人員不懂技術，也不想學
- 解決方案：**Email-First 整合**
  - 步驟 1：連結 Google 帳號
  - 步驟 2：允許 Gmail 存取
  - 步驟 3：完成
- 結果：90% 的用戶在 3 分鐘內完成設定

**實踐案例：** 2026-02-10 架構決策轉折

| 特性 | API 方案 | Email 方案 |
|------|---------|-----------|
| 設定步驟 | 5+ 步驟 | 3 步驟 |
| 審核時間 | 數週到數月 | 無需審核 |
| 技術門檻 | 需要了解 OAuth | 一般使用者即可 |
| 用戶完成率 | < 10% | > 80% |

**詳細記錄：** 請參閱 [docs/2026-02-09-steve-jobs-product-analysis.md](./docs/2026-02-09-steve-jobs-product-analysis.md)

---

#### 原則 2：技術隱形，價值顯現

**我們不會：**
- ❌ 在 UI 上顯示「我們使用 Gmail API v1」
- ❌ 要求用戶了解 OAuth 流程
- ❌ 顯示技術錯誤訊息（如「401 Unauthorized」）
- ❌ 讓用戶選擇「API 版本」或「同步頻率」

**我們會：**
- ✅ 顯示「✓ 已連結 Google 帳號，每 6 小時自動同步」
- ✅ 用自然語言說明狀態（「剛剛同步」而非「2026-02-10T14:30:00Z」）
- ✅ 提供一鍵解決方案（「重新連結」而非「檢查 access token」）
- ✅ 預設最佳設定，用戶無需選擇

**UI 設計原則：**
```
❌ 技術導向的 UI：
┌─────────────────────────────┐
│ LinkedIn API Configuration   │
│ [ ] Enable OAuth 2.0         │
│ [ ] Grant w_member_social   │
│ [ ] Set webhook endpoint    │
│ [ ] Configure sync interval │
└─────────────────────────────┘

✅ 用戶導向的 UI：
┌─────────────────────────────┐
│ 📬 社交媒體整合              │
│                             │
│ 連結 Google 帳號，自動同步    │
│ LinkedIn & Facebook 通知    │
│                             │
│ [連結我的 Google 帳號]       │
└─────────────────────────────┘
```

---

#### 原則 3：預設智慧，而非要求用戶做決定

**問題：** 應該讓用戶選擇同步頻率嗎？

**Android 思維：**
```typescript
<select>
  <option>每 1 小時</option>
  <option>每 6 小時</option>
  <option>每 12 小時</option>
  <option>每天</option>
</select>
```

**Steve Jobs 思維：**
```typescript
// 不給選項，直接用最佳設定
const SYNC_INTERVAL = 6 * 60 * 60 * 1000; // 6 小時
```

**理由：**
- 用戶不知道什麼是「最佳」
- 給選項會增加認知負擔
- 預設值 = 經過測試的最佳值
- 如果真的需要調整，進階選項藏在設定深處

---

#### 原則 4：錯誤處理是 UX 的一部分

**Android 思維：**
```typescript
catch (error) {
  console.error(error);
  alert('Error: LinkedIn API returned 401 Unauthorized');
}
```

**Steve Jobs 思維：**
```typescript
catch (error) {
  // 自動嘗試解決
  if (error.code === 'TOKEN_EXPIRED') {
    const refreshed = await googleService.refreshToken();
    if (refreshed) {
      return retryOperation();
    }
  }

  // 用戶友好的訊息
  showNotification({
    title: '連結已過期',
    message: '請重新連結您的 Google 帳號',
    action: () => router.push('/profile?tab=social')
  });
}
```

**差異：**
- Android：告訴用戶「壞了」
- Steve Jobs：告訴用戶「如何修復」，並自動嘗試修復
